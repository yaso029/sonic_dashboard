# Sonic CRM — UAE Accounting Firm CRM

Sonic CRM is a single-firm client-relationship and practice-management tool built
for a **UAE accounting firm**. It manages the full client lifecycle — lead intake,
client onboarding, engagements (bookkeeping, VAT, corporate tax, payroll, audit,
CFO, etc.), tasks, documents, UAE-focused invoicing (5% VAT, AED), recurring
retainers, a client self-service portal, and an optional AI document-analysis +
compliance-checklist layer.

> **Single-firm by design.** There is no `firm_id` / multi-tenancy. The whole app
> is one firm's instance and runs fully offline with no external services unless
> you opt in to Stripe (test mode) or Anthropic.

This system was refactored from an earlier real-estate CRM; the auth, dashboard,
HR, calendar, e-cards, notifications and partnerships scaffolding were retained,
and the entire property/listing/matching domain was replaced with the accounting
domain.

## Tech stack at a glance

**Backend** — Python, [FastAPI](https://fastapi.tiangolo.com/) 0.111, SQLAlchemy 2.0,
JWT auth (`python-jose`, HS256), passlib/bcrypt password hashing. SQLite by default
(local file `crm_local.db`), PostgreSQL in production.

**Frontend** — React 19 + [Vite](https://vite.dev/) 8, React Router 7, Tailwind CSS v4
(CSS-first config), axios, Recharts, `@hello-pangea/dnd` (Kanban), `react-hot-toast`,
`qrcode.react`.

**Optional integrations** — Stripe (test mode only, card payments), Anthropic Claude
(document analysis), Cloudinary (HR/e-card images only), Meta WhatsApp + webhooks
(legacy partnerships module).

See [ARCHITECTURE.md](ARCHITECTURE.md) for the full picture.

## Quickstart (local development)

### Prerequisites
- Python 3.11+ and `pip`
- Node.js 18+ and `npm`

### 1. Backend

```powershell
# from the repo root: D:\Sonic_crm\sonic-crm-main
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt

# optional: copy the env template (all values have safe local defaults)
copy .env.example .env

# run the API (auto-creates tables + seeds the admin on startup)
uvicorn backend.main:app --reload --port 8000
```

The API is now at **http://localhost:8000**, interactive docs at
**http://localhost:8000/docs**.

On first startup the app:
1. creates all tables (`Base.metadata.create_all`),
2. runs additive SQLite column migrations (`run_light_migrations`),
3. seeds the admin user, and
4. starts the background customer-sync scheduler.

### 2. Frontend

```powershell
cd frontend
npm install
npm run dev
```

The dev server prints a local URL (default **http://localhost:5173**). It talks to
the API at `http://localhost:8000` unless you set `VITE_API_URL`.

### 3. Log in

| Field | Value |
|-------|-------|
| Login name | `Yaso` |
| Password | `Yaso@123` |

> Staff log in by **full name**, not email (see [API_REFERENCE.md](API_REFERENCE.md)
> → `POST /api/auth/login`). The admin is seeded/re-seeded on every startup, so this
> account can never be permanently locked out. Change the password in production via
> Settings → password, and set a strong `SECRET_KEY`.

## Modules

**Accounting domain (the core of the refactor)**
- **Clients** — UAE entity profile (TRN, CT registration, trade licence, legal form, FY end, ESR flag), role-scoped visibility.
- **Services** — engagements per client (bookkeeping, VAT, corporate tax, payroll, audit, CFO, financial statements, company formation, tax consultation) with recurrence and fees.
- **Tasks** — work items, optionally linked to a client/service; standalone tasks allowed.
- **Documents** — client files on pluggable storage, signed-URL downloads, append-only access audit trail.
- **Billing** — UAE VAT invoices, manual + Stripe (test) payments, recurring subscriptions, revenue & AR-aging reports.
- **Client Portal** — separate login accounts for clients (invoices, pay, documents, service/task status, profile).
- **AI & Compliance** — deterministic UAE tax checklist (always on) + optional Claude document analysis.
- **Security** — security audit log + login lockout (admin viewer/escape hatch).

**Retained from the original platform**
- CRM leads (inquiry pipeline, Kanban, lead→client conversion), partnerships/outreach, HR, calendar, e-cards, customers/Meta sync, in-app + web-push notifications.

See [USER_GUIDE.md](USER_GUIDE.md) for how staff use each module and
[RBAC.md](RBAC.md) for who can do what.

## Verifying a change

Each backend phase ships an acceptance script under `backend/scripts/`. Run them
from the repo root with the venv Python:

```powershell
.\.venv\Scripts\python.exe -m backend.scripts.verify_phase3
.\.venv\Scripts\python.exe -m backend.scripts.verify_phase4
.\.venv\Scripts\python.exe -m backend.scripts.verify_phase5
.\.venv\Scripts\python.exe -m backend.scripts.verify_phase6
.\.venv\Scripts\python.exe -m backend.scripts.verify_phase7
.\.venv\Scripts\python.exe -m backend.scripts.verify_phase9
```

Frontend changes are gated by a clean production build:

```powershell
cd frontend
npx vite build
```
